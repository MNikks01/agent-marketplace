# Agent Marketplace — PROBLEM

## Core problem
**No trusted, monetized place to discover, install, and run AI agents/MCP servers/workflows.** Creators can't easily reach users or earn; teams can't easily find trustworthy, working capabilities.

## Pains
| Side | Pain | Cost |
|------|------|------|
| Consumers | Hard to find quality agents/servers; trust/safety unknown; integration friction | Reinvent the wheel; risk installing unsafe tools |
| Creators | No distribution; no monetization; no trust signal | Good work goes unseen/unpaid |
| Both | Fragmented (scattered repos, registries) | Inefficient ecosystem |

## Why existing tools fall short
- **GPT Store / plugin stores:** distribution but weak quality control + monetization + discovery.
- **MCP registries/directories:** list servers, but not curated/monetized/one-click-into-workflow.
- **Hugging Face:** great for models, not agent/workflow *commerce* with security review + revenue share.
The gap: a **curated, security-reviewed, monetized marketplace** integrated into a real workflow platform (#1), with revenue share. (See COMPETITOR_ANALYSIS.md.)

## Root cause (and why we build last)
Marketplaces require **simultaneous supply + demand + trust** — the cold-start problem. Most fail because they launch without distribution. We solve the root cause by *earning distribution + trust first* (via #1/#3/#4) and only then opening the marketplace.

## Validation
MCP registries appearing = discovery demand. Creators already publishing servers/agents (in scattered places) = supply latent. Teams asking "is there a ready-made agent for X?" = demand. GPT-Store's traction *and* quality problems show both the promise and the pitfall.

## Success
Creators publish and earn; teams discover trusted, security-reviewed capabilities and install them one-click into ContextOS workflows; usage-based revenue share keeps the flywheel turning.
